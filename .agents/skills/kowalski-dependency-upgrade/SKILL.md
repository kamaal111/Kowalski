# Kowalski Dependency Upgrade

## Overview

Load [dependency-upgrade-best-practices](../dependency-upgrade-best-practices/SKILL.md) first. Use this skill for Kowalski's dependency surfaces, generated artifacts, repo-standard commands, and project-specific breakage patterns.

## Upgrade Kowalski's Surfaces Deliberately

- Read [upgrade-surface.md](./references/upgrade-surface.md) for the current package map.
- Use the shared scripts in [dependency-upgrade-best-practices](../dependency-upgrade-best-practices/SKILL.md) to shortlist versions before editing manifests, and keep the default minimum age in [config.json](../dependency-upgrade-best-practices/config.json).
- Treat these manifests as the main edit surfaces:
  - root Node tooling in `package.json`
  - server Node dependencies in `server/package.json`
  - Swift package requirements in `app/*/Package.swift`
- Treat these files as generated outputs that should be regenerated, not edited directly:
  - `pnpm-lock.yaml`
  - `server/pnpm-lock.yaml`
  - `app/*/Package.resolved`
  - `app/Kowalski.xcodeproj/project.xcworkspace/xcshareddata/swiftpm/Package.resolved`
  - `app/KowalskiClient/Sources/KowalskiClient/openapi.yaml`

## Check Security Before Upgrading

- Investigate current compromise signals before selecting a version.
- If a package or version is suspected compromised, stop and clearly tell the user which package and version is affected, how it may have been compromised, and why that version should be avoided.

## Validate The Right Kowalski Surface

- After editing `package.json`, run the appropriate `pnpm install` flow so the lockfile is regenerated by pnpm.
- After editing `Package.swift`, run `swift build` in the affected package directory so SwiftPM resolves and regenerates `Package.resolved`.
- Run `just lint`, `just format-check`, and `just typecheck` for root-tooling-only changes as needed.
- Run `just compile-server` and `just test` for server-only dependency changes.
- Run `swift build` in each affected package directory, then `just test`, for Swift-only dependency changes.
- Run `just download-spec` when API or generator changes require refreshing the generated client input.
- Run `just ready` from the repository root as the final gate.

## Watch For Kowalski-Specific Breakage Patterns

- Expect `@hono/*`, `zod`, `drizzle`, or auth upgrades to ripple into route contracts, handler types, validation, and integration tests.
- Expect OpenAPI generator or runtime upgrades to affect both `KowalskiClient` output and feature call sites in `KowalskiFeatures`.
- Expect `ForexKit` or `KamaalSwift` upgrades to ripple through `KowalskiClient`, `KowalskiDesignSystem`, and `KowalskiFeatures`.
- Expect Swift upgrades to fail on warnings because packages use strict concurrency and warnings-as-errors settings.
